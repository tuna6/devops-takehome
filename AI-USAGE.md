# AI Usage Disclosure — Index

Workflow across all parts: For each part of the assignment, the implementation process followed a two-step pattern. First, architecture and design decisions were discussed in a Claude chat session — covering trade-offs, constraints specific to the local Docker-out-of-Docker environment, and potential failure modes — until a concrete plan was agreed on. Only then was that plan translated into a tightly-scoped Claude Code agent prompt, written with the chat session's context already resolved. This separation meant the agent received explicit, unambiguous instructions (environment facts, file paths, what not to do) rather than being left to infer them — the primary source of silent wrong assumptions in earlier runs. The chat session also served as the post-run auditor: agent output was reviewed here before being accepted, with real command output required as evidence.

This file indexes all session-level AI usage disclosures for this project. Each entry links to the detailed log in [`ai-usage/`](ai-usage/).

**Sample prompt — Claude chat session (audit pass before submission):**

> Before doing any new work, I need you to AUDIT existing code/config that was generated by a
> weaker model (Claude Haiku) in a previous session, for a DevOps take-home assignment.
>
> Read /path/to/ASSIGNMENT.md fully first — this is the assignment spec with exact, non-negotiable
> requirements. Do not edit this file, it's read-only reference.
>
> Then inventory and review everything currently in the repo against that spec. I do NOT want you
> to just say "looks fine" — I want a structured findings report, then we'll decide together what
> to fix, keep, or throw out.
>
> ## What to check specifically
>
> For whatever currently exists in docker-compose.yml, toolbox/, scripts/, or any other files:
>
> 1. **Architecture correctness**:
>    - Does the toolbox container use Docker-out-of-Docker (bind-mounted /var/run/docker.sock,
>      talking to the HOST's Docker daemon) — or did it incorrectly try to run a nested Docker
>      daemon (DinD), or install k3d/kubectl/etc. assuming they're on the reviewer's host?
>    - Does `docker compose up -d` alone (no manual steps) actually produce a working k3d cluster
>      with 1 server + at least 4 AGENT (worker) nodes? Don't trust comments claiming this — actually
>      run it and verify with `kubectl get nodes`.
>    - Are tool versions (k3d, kubectl, helm, terraform, k6) pinned to specific versions, or left
>      as "latest"/unpinned? If pinned, are the pinned versions actually real, current, valid
>      releases — verify against the actual GitHub releases/docs, don't assume Haiku's version
>      numbers are real or current.
>
> 2. **Idempotency**: Actually run `docker compose up -d` TWICE in a row and run
>    scripts/00-bootstrap-cluster.sh twice. Does the second run error, create a duplicate cluster,
>    or fail in any way? Show me real output, not assumptions.
>
> 3. **Hallucination check**: Haiku models are more prone to confidently inventing plausible-looking
>    but wrong flags, APIs, or config keys. Specifically verify:
>    - Every k3d CLI flag used actually exists in the pinned k3d version (check --help or docs,
>      don't assume).
>    - Every docker-compose.yml key/structure is valid Compose v2 syntax.
>    - Any referenced file paths, mounts, or scripts actually exist and match what's referenced.
>
> 4. **Scope creep check**: Did it build anything beyond cluster bootstrap (e.g. did it try to
>    write Part 1's app, a Helm chart, README.md, or other documentation prematurely)? Flag this —
>    I want to know what exists so we can decide whether to keep, defer, or discard it.
>
> 5. **Silent error handling**: Look for any `|| true`, swallowed exit codes, or `set +e` patterns
>    that might be hiding real failures rather than handling them — Haiku sometimes does this to
>    make scripts "appear" to succeed.
>
> ## Output format
>
> Give me a findings report structured as:
> - ✅ What's actually correct and verified working (with the command output that proves it)
> - ⚠️ What's questionable, unverified, or you're not fully confident about
> - ❌ What's actually broken, wrong, or hallucinated (with specifics — what's wrong and what it
>   should be instead)
> - 🗑️ What's out of scope / premature and should probably be removed or deferred
>
> Do NOT fix anything yet. Just audit and report. We'll decide what to do about each finding together
> after I see the report.
