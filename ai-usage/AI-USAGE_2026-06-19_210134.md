# AI Usage Disclosure

## Tools Used

| Tool | Role |
|---|---|
| **Claude (Sonnet, chat)** | Architecture planning, environment setup guidance (WSL2/Docker/GHCR), prompt engineering for Claude Code, and independent review/audit of AI-generated infra config |
| **Claude Code** | Implementation — scaffolding `docker-compose.yml`, `toolbox/Dockerfile`, and cluster bootstrap scripts. This disclosure covers the cluster-bootstrap foundation of the project (the Docker-out-of-Docker toolbox pattern and local k3d cluster provisioning) — i.e. the piece that has to work before any later part (app, Helm chart, ArgoCD, troubleshooting, CI/CD, load testing) can be evaluated at all.

## Representative Prompts

**0. Correcting a flawed architecture assumption before any code was written**

During planning (Claude chat, before Claude Code touched anything), Claude's first proposed bootstrap design implicitly assumed k3d would be installed and run directly **on the reviewer's host** — i.e. that `k3d cluster create` would be invoked from outside any container. This directly contradicts the assignment's Golden Rule, which guarantees the reviewer's machine has *only* Docker + Compose installed, nothing else.

This was caught and corrected before any implementation prompt was written:

> "But the rules requires just run docker compose up -d. Are you sure the interviewer environment has k3d installed? What if they want us to automatically install k3d?"

This forced a redesign to the actual architecture used: a self-contained "toolbox" container with k3d (and kubectl/helm/terraform/k6) baked into its own image, talking to the **host's** Docker daemon via a bind-mounted socket (Docker-out-of-Docker), so the reviewer's host never needs anything beyond Docker itself. Every later prompt to Claude Code was built on this corrected model — had this assumption gone unchecked, the entire bootstrap layer would have been built on a foundation that fails the assignment's first and most basic gate (`docker compose up -d` working with nothing else installed).

**1. Scoping the first implementation pass**

> "Right now I only want you to do ONE thing: scaffold the repo structure and get a working local Kubernetes cluster bootstrapped via `docker compose up -d`. Do NOT build the application (Part 1), Helm charts, ArgoCD, README.md, or anything else yet... Do not proceed to Part 1, Helm, ArgoCD, README.md, or anything beyond cluster bootstrap."

This deliberately constrained Claude Code to a single, verifiable slice instead of letting an agentic tool generate the whole repo in one pass — easier to review, easier to catch a bad assumption before it propagates into later parts.

**2. Auditing a weaker model's output before building on it**

> "Before doing any new work, I need you to AUDIT existing code/config that was generated by a weaker model (Claude Haiku)... I do NOT want you to just say 'looks fine' — I want a structured findings report... Don't trust comments claiming this — actually run it and verify... Haiku models are more prone to confidently inventing plausible-looking but wrong flags, APIs, or config keys."

This caught three fatal, build/runtime-breaking bugs in the original output (detailed below) that were not obvious from reading the code alone.

**3. Fixing a portability issue caught during review**

> "We need to remove `network_mode: host` from docker-compose.yml — it only works on Linux, and the reviewer's machine might be macOS... I want ONE fix that works everywhere, not an OS-conditional branch unless that's genuinely unavoidable."

## Example Where AI Was Wrong — How It Was Caught, What Was Done

### The audit's own findings (Haiku output corrected by a stronger-model review pass)

An initial implementation pass (run on Claude Code defaulting to Haiku, before this was noticed) produced a `docker-compose.yml` + bootstrap script that looked plausible on read-through but had **three fatal bugs**, surfaced only by actually running the system rather than reading the code:

1. **k6 download URL bug** — the version variable already included a `v` prefix, and the Dockerfile concatenated another literal `v` on top (`k6-vv2.0.0-linux-amd64.tar.gz`), producing a 404. This silently broke the entire `docker compose up -d` flow at image-build time.
2. **Unreachable kubeconfig (DooD networking)** — `k3d kubeconfig get` wrote a server URL of `https://0.0.0.0:<port>`, which only resolves to the real API server if the calling container shares the host's network namespace. It didn't. Every subsequent `kubectl` call would have hung — a structural bug invisible from a code read, only caught by actually executing `kubectl get nodes` and watching it fail.
3. **Missing `export KUBECONFIG`** before invoking the provided `prepare.sh` — the subprocess had no kubeconfig context, and would fail outright under `set -euo pipefail`, aborting the whole bootstrap.

**How this was caught:** rather than accepting the first implementation, a second review pass was explicitly instructed to *audit* — read the spec fully, inventory what existed, and **actually run** the system twice (build, up, idempotent re-run) and report real command output, not assumptions. This is what surfaced all three issues, plus two minor ones (a duplicated `kubectl wait` block, Docker socket mounted `:ro` instead of `:rw`).

**What was done:** each fix was applied individually with required verification (real build logs, real `kubectl get nodes` output, two consecutive idempotent runs) before being accepted — not just "should work now."

### A second, independently-caught issue: AI's own fix was non-portable

After the kubeconfig networking bug was fixed using `network_mode: host`, this was verified working — but on reflection (not prompted by a tool failure) it was recognized as a **Linux-only fix**: Docker Desktop on macOS does not share a real host network namespace with containers (it runs inside a lightweight VM), so a Mac-based reviewer running the Golden Rule (`git clone && docker compose up -d`) would have silently failed at this step with no clear error pointing back to the cause.

**What was done:** rejected the Linux-only fix despite it "working," and required a cross-platform replacement. The accepted fix uses `extra_hosts: host.docker.internal: host-gateway` (Docker ≥20.10's `host-gateway` magic value resolves on Linux; `host.docker.internal` is natively available on Docker Desktop Mac/Windows) plus a `sed` rewrite of the kubeconfig's server URL — no OS-conditional branching required, verified on the actual dev/test environment (WSL2 Ubuntu → native Ubuntu) with real `kubectl get nodes` output before being accepted.

## What Was Independently Verified (Not Just Accepted)

- Every pinned tool version (k3d, kubectl, helm, terraform, k6) was checked for a real, resolvable download URL before being locked in — including pushing back on a model's "verified" claim that only checked for an HTTP 200, not that the artifact/version convention was actually correct.
- k6 was deliberately pinned to a more established stable release line rather than the newest major version offered by the AI, for lower compatibility risk in a one-shot environment.
- `docker compose config --quiet`, full image build (specifically re-checking the previously-broken k6 layer), `docker compose up -d`, and **two consecutive runs** of both `docker compose up -d` and the bootstrap script were required to prove idempotency — no duplicate clusters, no errors on re-run.
- Real `kubectl get nodes` output confirming all nodes `Ready` was required after each networking fix, rather than trusting the fix "by inspection."

## Summary

AI tools were used heavily for both implementation (Claude Code) and architectural reasoning/review (Claude chat). The workflow deliberately separated "generate" from "verify" at every layer — including the planning layer, not just the code layer.

The most consequential catch happened before any code existed at all: Claude's own first architecture proposal assumed k3d would be available on the reviewer's host, which contradicts the assignment's core constraint (Docker + Compose only, nothing else pre-installed). Questioning this assumption — rather than accepting a plausible-sounding plan from the AI doing the planning — is what produced the correct Docker-out-of-Docker design that every subsequent script and container actually relies on. A wrong foundation here would have invalidated everything built on top of it, regardless of how well the later implementation was executed.

Downstream of that, three further defects were caught by treating Claude Code's first implementation pass (which had silently defaulted to a weaker model) as something to audit rather than trust: a build-breaking typo-class bug invisible without running the build, a structural kubeconfig/networking assumption invisible without running `kubectl`, and a portability gap (Linux-only fix) caught through manual reasoning even after that fix had already been verified "working" on the primary dev/test platform. In every case, the standard applied was the same: real command output required before accepting a claim, whether the claim came from generated code or from the AI's own planning suggestions.
