## Representative Prompts

**1. Specifying the migration as intent, not file copy**

> "Migrate the INTENT, not the file. Some practices below are exactly what we are migrating away from — handle them accordingly and explain your decisions in MIGRATION-NOTES.md."

This framing was deliberate: asking an AI to "migrate a CI file" without this instruction tends to produce a structural transliteration of the source (GitLab `stages` → GitHub `jobs`, each legacy step copied). Specifying "migrate the intent" forces the model to evaluate each legacy practice before carrying it forward.

**2. Making each legacy failure an explicit named item**

> "Do not copy any of the following forward: 1. Hardcoded AWS credentials… 2. `sonar_check` has `allow_failure: true`… 3. `unit_test` swallows failures with `|| echo`… 4. `docker push "...:latest"`… 5. `node:16` base for a Python service… 6. `docker:20.10-dind`… 7. `deploy_prod` doing `kubectl set image`"

Numbering the issues and quoting the exact YAML removed ambiguity about what "the bad practices" referred to. Without this, items 3 and 5 (failure-swallowing and wrong base image) would likely have been missed as subtler than the more obvious credential and tag issues.

**3. Requiring explicit loop-prevention documentation**

> "State this explicitly in `MIGRATION-NOTES.md` — don't rely on it silently."

Forces the loop-prevention reasoning (paths filter + `[skip ci]`) into the documentation rather than leaving it as implicit behaviour that the next engineer would have to reverse-engineer.

## Mistakes Made — Pre-flight (caught before any code ran)

### 1. Wrong repo root path

**What happened:** Initial file reads targeted `ASSIGNMENT.md` and `helm/quote-api/values.yaml` at `/home/tu/projects/elsa-devops-assignment/` (the outer checkout directory) instead of the actual git repo root at `devops-takehome/`.

**How caught:** Both reads returned "File does not exist." A `find` directory listing immediately showed the correct layout. Confirmed by `git remote -v` — the remote is `tuna6/devops-takehome.git`, so `devops-takehome/` is the repo root.

**Impact:** Caught before writing any files. Zero downstream effect.

---

### 2. Image name expression with hardcoded component

**What was drafted:** `ghcr.io/${{ github.repository_owner }}/devops-takehome` — owner variable plus hardcoded image name.

**What was wrong:** `github.repository` already encodes `owner/repo-name` (`tuna6/devops-takehome`), making the hardcoded `devops-takehome` redundant and fragile if the repo is renamed.

**How caught:** Cross-checking against the Part 1 GHCR image URL (`ghcr.io/tuna6/devops-takehome:2c94e39`) made the self-contained expression obvious.

**What was done:** Changed to `ghcr.io/${{ github.repository }}:${{ github.sha }}` throughout.

## Mistakes Made — Runtime (only visible after triggering CI)

### 3. Pinned Trivy action version that doesn't exist

**What Claude wrote:** `uses: aquasecurity/trivy-action@0.28.0`

**What happened:** GitHub Actions failed to resolve the action — tag `0.28.0` does not exist on the `aquasecurity/trivy-action` repository.

**How caught:** User reported the exact error from the failed workflow run: `Unable to resolve action aquasecurity/trivy-action@0.28.0, unable to find version 0.28.0`.

**What was done:** Changed to `aquasecurity/trivy-action@master`. This is a known-bad pattern (floating `master` tag is not reproducible) but is the pragmatic fix when the correct pinned version is unknown.

**Root cause:** Claude's training data has a cutoff before this action reached the version number it assumed. The assumption was not verified against the actual available tags.

---

### 4. Pinned starlette 0.41.3 had three HIGH CVEs — Trivy caught them

**What happened:** Trivy scanned the built image and found 3 HIGH vulnerabilities in `starlette==0.41.3`:

| CVE | Fixed in |
|---|---|
| CVE-2025-62727 — DoS via Range header merging | 0.49.1 |
| CVE-2026-48818 — SSRF via UNC paths in StaticFiles | 1.1.0 |
| CVE-2026-54283 — DoS via form() limits ignored | 1.3.1 |

With `exit-code: 1` and `ignore-unfixed: true`, the scan failed and blocked the push — which is exactly what a security gate should do.

**What Claude had assumed:** The pre-flight review noted "no patterns that well-written Semgrep rules flag as security issues" — but that review was for source code (Semgrep), not image dependencies (Trivy). The pinned `starlette==0.41.3` in `src/requirements.txt` was not checked against the CVE database before the workflow was written.

**How caught:** Trivy caught it live in CI. The scan output identified the exact package, installed version, and fixed versions.

**What was done:**
1. Temporarily un-pinned direct deps in `src/requirements.txt`
2. Rebuilt the Docker builder stage locally to let pip resolve the latest compatible versions
3. Captured `pip freeze` output from the builder container: `fastapi==0.138.0`, `starlette==1.3.1`, `uvicorn==0.49.0`, `prometheus_client==0.25.0`
4. Re-pinned `src/requirements.txt` with those versions — all three CVEs now fixed

**Lesson:** Pinned transitive deps that were safe at pin time can accumulate CVEs. Trivy is the correct tool to catch these; it did its job. The pre-flight code review cannot substitute for an image scan.

---

### 5. `permission_denied: write_package` on GHCR push — two steps needed

**What happened:** The push to GHCR failed with `denied: permission_denied: write_package` even though the workflow declared `permissions: packages: write`.

**First fix attempted (insufficient):** Changed repo **Settings → Actions → General → Workflow permissions** to "Read and write permissions". Re-run still failed with the same error.

**Root cause:** The `devops-takehome` package on GHCR was first created by the Part 1 manual push (`scripts/10-build-push.sh`) using the user's credentials directly. Packages created this way are not automatically linked to the repository, so `GITHUB_TOKEN` — which is scoped to the repository — does not have write access to the package.

**Actual fix:** Package settings → **Manage Actions access** → Add `tuna6/devops-takehome` repository with **Write** role. This explicitly links the package to the repo so `GITHUB_TOKEN` can push to it.

**What Claude missed:** The first fix suggestion (workflow permissions setting) was correct but incomplete. It did not account for the package-level access control that governs `GITHUB_TOKEN` push permissions for packages created outside CI. The second fix was only identified after the user reported that re-running after the first fix still failed.

## Summary

Three mistakes only surfaced at runtime, not during pre-flight review:

1. **Trivy action version** — hallucinated a tag that doesn't exist. Caught immediately from the workflow error. Fix: `@master`.
2. **Starlette CVEs** — pinned dep was vulnerable; not caught by source review but correctly caught by Trivy doing its job. Fix: upgrade fastapi + starlette to patched versions via pip freeze re-capture.
3. **GHCR push permission** — two-step fix required (workflow permissions setting + package-level Actions access); the first fix alone was insufficient. The second step was not anticipated in the initial setup instructions.

All three were caught and fixed before the final green run. The Trivy CVE finding is the most significant: it demonstrates the Trivy gate doing real work on real vulnerabilities, not just running as a formality.
