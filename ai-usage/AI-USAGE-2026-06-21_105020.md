## Representative Prompts

**1. Specifying the migration as intent, not file copy**

> "Migrate the INTENT, not the file. Some practices below are exactly what we are migrating away from — handle them accordingly and explain your decisions in MIGRATION-NOTES.md."

This framing was deliberate: asking an AI to "migrate a CI file" without this instruction tends to produce a structural transliteration of the source (GitLab `stages` → GitHub `jobs`, each legacy step copied). Specifying "migrate the intent" forces the model to evaluate each legacy practice before carrying it forward.

**2. Making each legacy failure an explicit named item**

> "Do not copy any of the following forward: 1. Hardcoded AWS credentials… 2. `sonar_check` has `allow_failure: true`… 3. `unit_test` swallows failures with `|| echo`… 4. `docker push "...:latest"`… 5. `node:16` base for a Python service… 6. `docker:20.10-dind`… 7. `deploy_prod` doing `kubectl set image`"

Numbering the issues and quoting the exact YAML removed ambiguity about what "the bad practices" referred to. Without this, the AI might have missed items 3 and 5 (the failure-swallowing and wrong base image) because they are subtler than the more obvious credential and tag issues.

**3. Requiring explicit loop-prevention documentation, not silent implementation**

> "This `paths` scope is deliberate: it's the mechanism that prevents the Git write-back step from re-triggering the workflow in an infinite loop, since that step only touches `helm/quote-api/values.yaml`, which falls outside `src/**` and `Dockerfile`. State this explicitly in `MIGRATION-NOTES.md` — don't rely on it silently."

The "state this explicitly" instruction matters: a CI engineer reading the paths filter in isolation might not understand it is a loop-prevention mechanism, not just a scope optimisation. The prompt forced the explanation into MIGRATION-NOTES.md rather than leaving it as implicit institutional knowledge.

## Real Mistake Made and Caught During This Work

### Working directory confusion: tried to read files at the wrong repo root

**What happened:** The initial file reads targeted `ASSIGNMENT.md` and `helm/quote-api/values.yaml` at `/home/tu/projects/elsa-devops-assignment/` — the outer directory — rather than `/home/tu/projects/elsa-devops-assignment/devops-takehome/`, which is the actual git repository root.

**Why this happened:** The task description used paths like `helm/quote-api/values.yaml` without a qualifying prefix, and the working directory context (`/home/tu/projects/elsa-devops-assignment`) matched the outer directory. The AI started reads at that path.

**How it was caught:** Both reads returned "File does not exist." — the outer directory only contains `ci/legacy.gitlab-ci.yml` and `devops-takehome/`. A directory listing (`find` command) immediately showed the actual layout: all project files live under `devops-takehome/`, which is also confirmed by the git remote (`git@github.com:tuna6/devops-takehome.git`) — the repo root is `devops-takehome/`, not the checkout parent.

**What was done:** All subsequent reads and file writes target the correct paths under `devops-takehome/`. The workflow is at `.github/workflows/ci.yml` relative to that directory, which is where GitHub Actions expects it (`.github/` at the repository root). If the workflow had been written to the wrong path, it would have been silently ignored — no error from GitHub, just no CI runs.

**Why this matters as a category of mistake:** In a nested repo layout (assignment zip → submission repo), the git remote is the ground truth for "repo root," not the checkout path on disk. Checking `git remote -v` and the path that ASSIGNMENT.md actually lives at disambiguates this immediately. The mistake was caught at the first file read — before any code was written — so it had no downstream consequence.

### Second issue caught: image name expression choice

**What was initially drafted:** `ghcr.io/${{ github.repository_owner }}/devops-takehome` — combining the owner variable with a hardcoded image name.

**What was wrong:** `github.repository_owner` is `tuna6`. Hardcoding `devops-takehome` separately means the expression would break if the repository were renamed, and it reads as if "devops-takehome" is a separately chosen image name rather than the repo name. Additionally, the expression is slightly longer and more fragile.

**How it was caught:** When verifying the GHCR image URL from Part 1 (`ghcr.io/tuna6/devops-takehome:2c94e39`), it's clear the image path is `owner/repo-name` — exactly what `github.repository` already encodes (`tuna6/devops-takehome`). Using `ghcr.io/${{ github.repository }}` is self-contained: no hardcoded component, and if the repo is renamed, the expression still resolves correctly.

**What was done:** Changed to `ghcr.io/${{ github.repository }}:${{ github.sha }}` throughout.
