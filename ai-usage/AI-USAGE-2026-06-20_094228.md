## Representative Prompts

**1. Scoping the implementation with mandatory real-output verification**

> "SCOPE — ONLY do the following. Do NOT touch Helm charts, ArgoCD, README.md... VERIFICATION — do not report this as done until you've actually run it and show me real output, not assumptions: Build the image locally and show the final size... Time a single /api/quote request and confirm it's in the ~100ms ballpark — paste the real timing, not an estimate."

Same pattern as Part 0: constrain scope tightly, and require the agent to produce actual command output (curl responses, timings, `whoami`) rather than asserting something works.

**2. Pushing back on a build-script architecture that would break for the reviewer**

> "When the reviewer run the test, they don't have the credential to push to ghcr? Why they required it in the assignment"

This surfaced a real design problem: the GHCR push step is a one-time *publishing* action the developer does before submission, not something that belongs in the reviewer's required path (`run-all.sh`). The fix was to keep `scripts/10-build-push.sh` as a dev-only utility, explicitly excluded from the reviewer's script sequence, with deployment manifests referencing the already-published image by SHA tag instead.

**3. Requiring full re-verification after a base-image change, not just a size check**

> "Before I accept the Alpine switch — please re-run the full verification suite against the NEW Alpine-based image, not just confirm it builds... size alone isn't sufficient verification for a base image switch."

## Example Where AI Was Wrong (or Initially Under-Justified) — How It Was Caught, What Was Done

### Base image choice: distroless → slim → Alpine, in two corrected steps

**First pass (agent's initial Dockerfile):** used `python:3.12-slim` (249 MB), explicitly rejecting `gcr.io/distroless/python3` with the stated reason that it ships Python 3.11 only and that `uvicorn[standard]`'s compiled extensions (`httptools`, `uvloop`) "need glibc compat we'd have to re-package."

**What was questioned:** the glibc-compat claim was about distroless specifically, but it left an unexamined assumption — that compiled extensions also rule out Alpine (musl libc), which is a separate question from distroless's Python version problem. Rather than accept "slim because distroless doesn't work," the agent was asked to re-check Alpine directly.

**What the agent found on re-investigation:** the original glibc concern didn't actually hold for Alpine — `httptools`, `uvloop`, `pydantic-core`, and `watchfiles` all publish prebuilt **musllinux** wheels on PyPI, so no compiler toolchain was needed in the builder stage either way. Final image dropped from 249 MB to **142 MB** with no added build complexity.

**Why this mattered enough to re-verify rather than just accept the smaller number:** a base-image swap changes the C library underneath the entire runtime (musl vs glibc), which can affect behavior in ways that don't show up in a "does it build" check — DNS resolution differences, and any transitive dependency that falls back to compiling from source instead of using a prebuilt wheel. Accepting "142 MB and it builds" without re-running the same verification used on the first image would have been exactly the kind of unverified AI output the assignment explicitly says it's screening for.

**What was done:** required the agent to re-run the *entire* original verification suite against the new Alpine image specifically, not just confirm the build succeeded:
- All four endpoints curled with real responses pasted (`/healthz`, `/readyz`, `/api/quote`, `/metrics`)
- `/api/quote` timed across 5 requests again (99–102ms, consistent with the pre-switch 101.7ms baseline — confirming musl's libc differences didn't silently change the CPU-bound timing characteristics)
- `/metrics` counter re-confirmed incrementing correctly (1→4 after 3 hits)
- `whoami` re-confirmed non-root (`appuser`, UID 1001) on the actual image that ships

Only after all five came back with real pasted output was the Alpine image accepted as final.

### A second issue: GHCR push wired into the reviewer's required path

**The problem:** the build/push script (`scripts/10-build-push.sh`) tags and pushes to GHCR using the developer's own credentials. If invoked as part of `run-all.sh` — the exact command the README's "Golden Rule" says the reviewer will run — it would fail for the reviewer, who has no GHCR write credentials and was never meant to need any.

**Root cause once examined:** the README requirement ("image pushed to GHCR, tagged with git SHA") describes a *publishing* step the developer performs once before submission, not a step that re-runs in the reviewer's environment. The reviewer is expected to **pull** the already-published public image, not rebuild and push a new one.

**What was done:** confirmed the build/push script needs to be excluded from `run-all.sh`'s script sequence and treated as a dev-only utility; Part 2's Helm chart will reference the image by its already-pushed SHA tag rather than triggering a build. Also flagged, as a follow-up check rather than an assumption: GHCR packages default to **private** even when the source repo is public — this needs explicit verification in the package's GHCR settings before submission, since a private package would silently 403 the reviewer's pull and look identical to a credentials error.

## What Was Independently Verified (Not Just Accepted)

- Real `curl` output for all four endpoints, on both the pre- and post-Alpine images — not just "should return 200."
- `/api/quote` timing measured across 5 real requests on each image build (101.7ms slim, 99–102ms Alpine), not estimated — used to confirm the SHA-256 CPU-burn implementation behaves consistently across a base image change.
- `/metrics` counter checked for actual increment behavior (value before vs. after N requests), not just presence of the metric name in output.
- Non-root execution confirmed via real `whoami` inside the running container on the final shipped image, not inferred from the Dockerfile's `USER` directive alone.
- Wheel availability (musllinux vs manylinux) checked directly against what the actual `pip install` pulled, rather than trusting the agent's first-pass claim that compiled extensions required glibc.
- The build/push script's placement was checked against the README's literal "Golden Rule" reviewer flow (`docker compose up -d && ./scripts/run-all.sh`) rather than assumed correct because it ran successfully in the developer's own environment.

---

## Summary

Part 1 followed the same generate → verify discipline established in Part 0: every agent claim (image size, timing, non-root, metrics) required real pasted command output before being accepted, not assertions. Two genuine issues were caught this way — an initially under-justified base-image rejection (distroless ruled out correctly, but Alpine ruled out by an assumption that didn't hold, costing 107 MB until questioned) and a script architecture that would have broken the assignment's stated reviewer flow had it shipped unexamined. Both were caught by treating "it works on my machine" and "it builds" as insufficient evidence on their own.
