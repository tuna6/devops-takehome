## Example Where AI Was Suboptimal — How It Was Caught, What Was Done

When asked for the NetworkPolicy allow-rules, the first draft from Claude scoped the egress rule's DNS destination using a generic `kube-system` namespace selector without first confirming the actual label key the cluster uses to identify that namespace (`kubernetes.io/metadata.name` vs. some other convention some clusters use, e.g. a custom `name` label). It also assumed the smoke-test client pod carries the label `app: smoke-client` without that being verified against the real file.

**How this was caught:** before applying, I opened `troubleshoot/smoke-job.yaml` myself and checked the actual pod labels and the namespace metadata, rather than trusting the AI's selector assumptions. The `app: smoke-client` label happened to be correct, but I verified it rather than assuming it — this is exactly the kind of small, plausible-but-unverified assumption that breaks a NetworkPolicy silently (an `egress` rule with a selector that matches zero pods fails closed, with no error, which would have produced the same symptom — DNS/connection failures — as the original bug, making it easy to mistake "my allow-rule is wrong" for "the policy needs another rule" and chase a ghost.

**What was done:** verified the label selectors against the actual `smoke-job.yaml` content before applying the policy, then re-ran `verify.sh` and confirmed all 7 checks passed with real output (pasted in `TROUBLESHOOTING.md`) rather than accepting the policy as correct because it was syntactically valid YAML.

## What Was Independently Verified (Not Just Accepted)

- The NetworkPolicy's pod/namespace selectors were checked against `troubleshoot/smoke-job.yaml`'s actual labels before applying — not assumed from the AI's draft.
- `verify.sh` was re-run after the NetworkPolicy fix and required to print `PASS` on all 7 checks before the fix was considered done; the real output is included in `TROUBLESHOOTING.md`.
- All six other issues (GPU toleration/nodeSelector, oversized memory request, ConfigMap name mismatch, invalid image tag, wrong probe port, Service selector/targetPort mismatch) were diagnosed via `kubectl describe`/`kubectl get -o yaml` and fixed without AI input — no AI claim to verify there, since none was made.

## Summary

Part 3 was almost entirely manual: reading `kubectl describe` output, identifying root causes, and fixing the manifest directly accounted for 6 of 7 issues. AI was used narrowly for the last issue (NetworkPolicy allow-rules) where it produced a plausible draft containing an unverified label-selector assumption — caught by checking the referenced file directly before applying, rather than trusting the policy because it was well-formed. AI was also used to turn raw diagnosis notes and terminal logs into `TROUBLESHOOTING.md`, with an explicit instruction to represent the manual/AI-assisted split accurately rather than overstate AI's role.
