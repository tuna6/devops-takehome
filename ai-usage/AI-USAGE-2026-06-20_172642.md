The prompt pre-answered the placement trade-off question before the AI could apply its defaults. ## Examples Where the AI Was Initially Wrong — How Each Was Caught, What Was Done

### 1. First draft used the wrong topology key label

**What the AI produced (internal first draft):** `topologyKey: node-type` and `key: node-type` in the affinity preference, without the `acme.io/` namespace prefix.

**What was wrong:** the actual node labels applied by `troubleshoot/prepare.sh` are `acme.io/capacity` and `acme.io/node-type` — namespaced. A bare `node-type` label does not exist on any cluster node. The topology spread with `topologyKey: node-type` would see zero topology domains (no node has that label) and fall back to treating all nodes as one domain — completely negating the capacity-aware distribution.

**How it was caught:** not by running the broken code. Caught before implementation by re-reading `troubleshoot/prepare.sh` to confirm the exact label keys used, then checking `kubectl get nodes --show-labels` for the running cluster. The `acme.io/` prefix is explicit in both.

**What was done:** used the correct fully-qualified label keys in the final implementation. `helm template` output confirmed they render into the Deployment spec as written.

### 2. Second draft used `DoNotSchedule` — would break the drill it was supposed to prove

**What the AI initially drafted (before being constrained by the prompt):** topology spread with `whenUnsatisfiable: DoNotSchedule`. The rationale offered was: "this guarantees at least one pod in each capacity domain."

**What was wrong:** with exactly 2 capacity topology domains (spot, on-demand) and 3 replicas at minAvailable, a `DoNotSchedule` constraint means: when the spot domain loses a node, Kubernetes must reschedule the evicted pod into one of the two remaining domains. But if both remaining schedulable nodes are in the *same* domain (e.g., both spot), the constraint blocks scheduling: placing another pod in spot would create a skew of 2 vs 1, which violates maxSkew:1 with `DoNotSchedule`. The replacement pod stays `Pending`. The PDB fires. The drill hangs.

This is specifically the scenario the drill is designed to test. The hard constraint that "guarantees the split" is the same hard constraint that breaks survivability under the reclaim scenario it was supposed to prove.

**How it was caught:** by reasoning through the drill scenario explicitly before running any code. The prompt pre-specified `ScheduleAnyway` and stated the reason. The AI's own first instinct was `DoNotSchedule` — the prompt forced the correct constraint.

**What was done:** used `ScheduleAnyway` for both topology spread constraints. The drill ran with 60 requests, 0 failures, max gap 2s, placement PASS. The `DoNotSchedule` version would have hung the drain step.

### 3. ArgoCD selfHeal reverted the placement test — required a workaround not planned for

**What happened:** after applying the Helm chart changes locally (not yet pushed to git), a `kubectl apply` of the rendered template triggered ArgoCD's selfHeal to immediately revert the Deployment to match the committed git state. The new pod (`7874ccb9db-27zlj`) on agent-0 went to `Error` state as ArgoCD killed it.

**Why this was a problem:** the assignment requires verifying real pod placement with actual output. With selfHeal active, any local apply is reverted within seconds — too fast to observe steady-state placement.

**What was done:** suspended ArgoCD automated sync (`kubectl patch application quote-api -n argocd --type merge -p '{"spec":{"syncPolicy":{"automated":null}}}'`), re-applied the chart, waited for the rolling update to complete, observed placement, ran the full drill, then re-enabled automated sync. ArgoCD will return to Synced+Healthy after the user pushes the deployment.yaml change to git.

This is not an AI error in the implementation — it's an operational detail of testing against a live ArgoCD instance. Documenting it here because it's the kind of non-obvious interaction that needs to be handled explicitly when testing GitOps-managed resources locally.

### 4. Weight:100 + soft topology spread does not reliably produce 2 spot / 1 on-demand

**What was claimed/expected:** the design — `preferredDuringScheduling` spot weight 100 plus `ScheduleAnyway` topology spread on `acme.io/capacity` — would "typically" yield a 2 spot / 1 on-demand split. The README's residual risk paragraph said "likely, not guaranteed."

**What was actually observed:** two different scheduling rounds in the same session produced different splits:

```
# ArgoCD-managed sync (fresh scheduling from 0 pods):
NAME                         NODE                      CAPACITY
quote-api-64cbcfdf46-qtjkv   k3d-elsa-devops-agent-0   spot
quote-api-64cbcfdf46-7xgkd   k3d-elsa-devops-agent-1   spot
quote-api-64cbcfdf46-6wvts   k3d-elsa-devops-agent-2   on-demand
→ 2 spot / 1 on-demand (PASS)

# Manual local apply (rolling update from existing pods):
NAME                       NODE                      CAPACITY
quote-api-9d75bc45-g44j5   k3d-elsa-devops-agent-0   spot  ← 2 pods same node
quote-api-9d75bc45-ns9fp   k3d-elsa-devops-agent-1   spot
quote-api-9d75bc45-sc9mw   k3d-elsa-devops-agent-0   spot
→ 3 spot / 0 on-demand (would print WARN at drill placement check)
```

The "3 spot / 0 on-demand" outcome did not appear once as a transient artifact — it reproduced across multiple scheduling rounds (delete-all-pods then reschedule, scale 0→3). The weight-100 spot preference consistently outscored the `ScheduleAnyway` topology spread penalty, keeping all pods on spot nodes. The "likely" framing in the README underestimated how strongly the preference weight suppresses the soft spread.

**How it was caught:** by actually running `kubectl get pods -o wide` after each scheduling event, not by reasoning about the expected outcome from the spec on paper. The 3/0 result appeared the first time pods were scheduled via local testing (manual apply with ArgoCD sync suspended). It was not caught during the first drill run because that drill happened to start from a 2/1 placement and the post-drain check showed PASS — the all-spot risk wasn't visible until the rolling update path was exercised.

**What was done — and what was deliberately not done:**

*Evaluated:* lowering the spot preference weight from 100 to ~20, which would give the topology spread's rebalancing scores (roughly 50–80 points for the under-represented domain) enough headroom to redirect a pod to on-demand. In limited testing this produced the 2/1 split more reliably.

*Rejected:* tuning the weight down was rejected because it changes the probability, not the guarantee. A weight of 20 might produce 2/1 most of the time on this specific 3-node test cluster, but the same constraints on a cluster with more spot nodes (or different relative scoring from other active plugins) could still produce any split. Tuning toward a demo outcome and presenting it as a reliable result would misrepresent what soft constraints actually do.

*Accepted:* documented the real observed range (2/1 under ArgoCD fresh scheduling; 3/0 under rolling-update local scheduling) in the README's residual risk paragraph, with real kubectl output as evidence. The drill's placement check explicitly handles this with a non-blocking WARN. This is the honest disclosure the assignment's verification standard asks for — "paste real command output, even if not 2/1."
