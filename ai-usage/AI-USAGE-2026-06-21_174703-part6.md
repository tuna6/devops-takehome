## Representative Prompts

**1. Requiring threshold justification, not generic numbers**

> "k6 `thresholds` block must use real numbers derived from your step 3 baseline... thresholds passing or failing honestly based on real run output — do not adjust thresholds after the fact to make them pass if the real numbers don't support it; report honestly if a chosen threshold doesn't hold and say why."

This set the standard before any threshold was written, not after — the same anti-gaming principle applied to the Part 4 SAST gate, applied here to a load-test gate.

**2. Pushing back on a threshold that passed too easily**

> "A threshold passing by a 10x margin isn't validating anything meaningful; it's the same 'a scan that can never fail is decoration, not a gate' problem the assignment calls out for the legacy CI pipeline in Part 4, just applied to a load test threshold instead of a SAST gate... tighten the threshold to something the actual data supports."

The first k6 threshold (`p(95)<2000ms`) passed at 178ms — over 10x margin. Rather than accept a passing threshold at face value, this was treated as a signal the threshold itself was wrong, not a result to celebrate.

**3. Rejecting an unverified root-cause claim**

> "This is a real, bigger claim — and so far it's based on 'node CPU usage looked low,' which is indirect... worth verifying directly rather than accepting the inference."

The agent attributed part of a latency discrepancy to "WSL2 doesn't enforce cgroup CPU quotas strictly," based only on `kubectl top nodes` looking low. This was not accepted without direct evidence (see below).

## Example Where AI Was Wrong — How It Was Caught, What Was Done

### A plausible-sounding root cause that direct measurement disproved

The first k6 threshold attempt (`p(95)<2000ms`) was derived from a Little's Law queuing estimate assuming each pod serves one request at a time (serial), predicting p95 ≈ 1500–1700ms at peak load. The real run measured p95 = 178.82ms — about 10x better than predicted.

Asked to reconcile the gap, the agent identified two contributing factors:
1. **Correct and verified**: the app's route handler is a sync `def`, which FastAPI runs via anyio's threadpool (~20 concurrent requests/pod, not 1) — directly explains most of the gap.
2. **Plausible but unverified**: "WSL2's CFS scheduler does not enforce `cpu.cfs_quota_us` as strictly as a native Linux kernel," offered as a secondary explanation, based only on `kubectl top nodes` showing low node-level usage during the test.

**How this was caught:** node-level `top` output is an indirect signal — it doesn't actually show whether a specific container's cgroup quota is being enforced. Rather than accept a kernel-behavior claim on indirect evidence, the pod's actual cgroup state was checked directly:

```
$ kubectl exec -n quote-api deploy/quote-api -- cat /sys/fs/cgroup/cpu.max
50000 100000

$ kubectl exec -n quote-api deploy/quote-api -- cat /sys/fs/cgroup/cpu.stat
nr_periods 47806
nr_throttled 5213
throttled_usec 225877632
```

This **disproved** the claim: the quota was correctly programmed (500m) and was actively being enforced — ~11% of all CPU periods were throttled, totaling ~226s of cumulative throttled time. WSL2 was not being lenient; the kernel was doing its job correctly.

**What was done:** the WSL2-leniency claim was removed from LOADTEST.md and the k6 script comments. It was replaced with the real, now-confirmed explanation: CPU throttling is real and measurable, but the app's thread-pool concurrency prevents it from manifesting as the kind of serial request-queue pileup the original Little's Law model assumed — it shows up as latency jitter under sustained load instead. The throttling data itself was kept and repurposed as direct evidence for LOADTEST.md's bottleneck analysis, rather than discarded once the original claim built on it was corrected.

The k6 threshold was also tightened from the 10x-loose `p(95)<2000ms` to `p(95)<400ms` (≈2.24x the observed 172–179ms across two runs), re-run and re-verified to still pass — a margin chosen to tolerate real run-to-run variance and cold-start jitter on scale-out, without being so loose it stops being a meaningful gate.

## What Was Independently Verified (Not Just Accepted)

- **ServiceMonitor scraping** was confirmed two independent ways: via a direct Prometheus API query (`up{job="quote-api"}` = 1 for all pod instances) and via all four Grafana dashboard panel PromQL queries each returning non-empty data from the same Prometheus datasource (checked via the Grafana API before the live run, not assumed from the import response alone) — confirming the result wasn't an artifact of one query path.
- **The HPA scale-out arithmetic** in an early LOADTEST.md draft (`ceil(3×264/70)=11.3`) was independently recalculated and found incorrect (`ceil(3×264/70)=12`) — a small, non-outcome-changing error, but corrected rather than left in an evidence-grade document.
- **The alert threshold margin** (30% vs. an observed 14% throttle ratio, claimed as "2.1x") was independently recomputed (`30/14 ≈ 2.14`) and confirmed accurate.
- **Screenshot timing** for the Grafana dashboard was checked against the k6 script's actual stage boundaries (cumulative time math, not isolated stage durations) before accepting it as evidence of "during the load test" — prompted by an earlier practice run where a manually-taken screenshot was discovered, after the fact, to have landed during the scale-down tail (replicas still elevated, but request rate already near zero), which would not have satisfied the requirement despite looking superficially correct.
- **`kubectl get pods -o wide` placement data** at peak scale-out (8 replicas) was checked against known node capacity labels rather than assuming a specific spot/on-demand ratio — the real result (7 spot, 1 on-demand, 0 GPU/control-plane) was reported as-is, consistent with the same honesty standard applied to Part 2's placement policy (which similarly produced an imperfect 3/0 split rather than a "clean" 2/1).

## Summary

Part 6's AI-assisted work followed the same generate/verify split as prior parts, with two notable additions: (1) a passing-but-meaningless threshold was treated as a problem to investigate, not a result to accept, and (2) a plausible secondary root-cause claim (WSL2 cgroup leniency) was checked against direct evidence and found to be wrong — the quota was being enforced correctly, and the real explanation for the original estimate's error was almost entirely the app's request-concurrency model. The corrected throttling data was kept and used productively in LOADTEST.md's bottleneck analysis rather than discarded once the surrounding claim was fixed.
