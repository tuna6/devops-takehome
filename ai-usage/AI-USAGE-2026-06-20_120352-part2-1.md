## Examples Where the AI Was Wrong — How Each Was Caught, What Was Done

### 1. ArgoCD version was stale and outside the support window

**What the AI produced:** `ARGOCD_VERSION=v2.13.3`, justified as "the most recent LTS-style minor line as of early 2025."

**What was wrong:** v2.13.x was already outside ArgoCD's supported window (project only patches the two most recent minors). The actual current stable line was v3.4.x. The AI had correctly pinned *a real version* — v2.13.3 does exist and installs — but it was the wrong version to ship.

**How it was caught:** The user flagged it by name in the follow-up prompt. The AI's justification ("most recent LTS-style minor") was reasoning from training-data memory of the release schedule, not from the actual current state of the project.

**What was done:** Fetched `https://github.com/argoproj/argo-cd/releases` directly via `WebFetch` and confirmed v3.4.4 as the current stable patch (released 2026-06-18). Did not assume a v3.x version number — confirmed it on the page before writing it into the script.

---

### 2. `kubectl apply` on ArgoCD v3 fails with a 262144-byte annotation limit error

**What the AI produced:** `kubectl apply -n argocd -f "${ARGOCD_MANIFEST}"` — standard client-side apply.

**What was wrong at runtime:**
```
The CustomResourceDefinition "applicationsets.argoproj.io" is invalid:
metadata.annotations: Too long: may not be more than 262144 bytes
```
ArgoCD v3's `applicationsets.argoproj.io` CRD is large enough that Kubernetes rejects the `kubectl.kubernetes.io/last-applied-configuration` annotation that client-side apply writes. The script exited with code 1 on the first real run against a fresh cluster.

**How it was caught:** The script actually ran. The error was not caught during planning or code review — it only surfaced at execution time.

**What was done:** Added `--server-side` to the apply. Server-side apply sends the manifest to the API server for field management instead of storing it in an annotation, bypassing the size limit entirely. Re-ran teardown → fresh cluster → deploy script → confirmed all resources created successfully. Also updated the comment in the script to explain why `--server-side` is required for v3, so the reason is on file.

---

### 3. emptyDir mounts were added speculatively, without evidence

**What the AI initially produced (session 1):** `volumeMounts` for `/app/__pycache__` and `/tmp`, with the justification: "Python tries to write `/app/__pycache__/main.cpython-312.pyc` at startup; without a writable mount it silently falls back to source-only import (no crash, but a warning in logs on every restart). Mounting an emptyDir here avoids the repeated cache-miss penalty."

**What was accepted too quickly:** The same session then reported "uvicorn started cleanly with no errors" as verification that the mounts were needed. This only showed the pod boots — it did not show that anything actually tried to write to those paths under real traffic.

**How it was caught:** The follow-up prompt correctly identified the gap: startup alone doesn't exercise write-on-first-use paths. The prompt required removing both mounts, redeploying, sending 30 real requests to `/api/quote`, and checking all three pod logs for `OSError`, `PermissionError`, or read-only filesystem messages.

**What the test showed:** Zero errors across all 3 pods after 30 requests through the Ingress. No Python warnings about `__pycache__`. No /tmp write attempts. Both emptyDir mounts were unnecessary — Python handles the bytecode cache miss silently (no log output), and uvicorn in single-worker mode with no `--reload` flag makes no runtime writes.

**What was done:** Removed both mounts permanently from the chart. The comment now reads with a verified negative finding rather than a speculative justification. Leaving unnecessary writable mounts on an otherwise-hardened pod (`readOnlyRootFilesystem: true`, `allowPrivilegeEscalation: false`, `capabilities: drop: ALL`) weakens the security posture without any actual benefit.

### 4. `run-all.sh` was broken for the reviewer's path — not caught until it was actually tested from the host

**What the AI initially produced:** `run-all.sh` called `"${SCRIPT_DIR}/00-bootstrap-cluster.sh"` directly. At face value this looks correct: the script exists, the path is right.

**What was wrong:** `k3d` and `kubectl` are not installed on the host — they exist only inside the toolbox container. Running `./scripts/run-all.sh` from the host (the exact Golden Rule command) would fail immediately with "k3d: command not found." The same test the assignment says the reviewer will run.

**Also wrong in `00-bootstrap-cluster.sh`:** The existing-cluster branch checked `k3d node list` for `< 5` containers and immediately errored out if fewer were found. This creates a race condition: if the reviewer runs `./scripts/run-all.sh` while the compose bootstrap container is still creating the cluster, the script sees the cluster exists but only 1–3 of 5 node containers registered yet, and exits with `ERROR`. The cluster wasn't broken — it was still starting.

**How both were caught:** The user asked "is the bootstrap script include wait? the reviewer might run ./run-all.sh while cluster is still in bootstrap state." This prompted reading both scripts carefully and testing `./scripts/run-all.sh` literally from the host — which would have failed.

**What was done:**
- `run-all.sh` rewritten to use `docker wait devops-takehome-bootstrap-1` (blocks until the bootstrap container exits) then `docker exec elsa-devops-toolbox` for each subsequent script. This means `docker` is the only tool required on the host, which any reviewer has by definition.
- `00-bootstrap-cluster.sh` existing-cluster branch changed from an immediate error to a polling wait (3-second intervals, 2-minute timeout), falling through to the existing `kubectl wait --for=condition=Ready nodes --all` gate. Genuine broken clusters still error — just after a timeout with a clear message, not immediately after a transient low node count.

Both fixes were verified by running `./scripts/run-all.sh` from the host against a live cluster and confirming exit 0 end-to-end.

## What Was Independently Verified (Not Just Accepted)

- **ArgoCD version:** confirmed v3.4.4 on `github.com/argoproj/argo-cd/releases` via live fetch — not asserted from training-data memory.
- **v3 RBAC impact:** read the actual `cluster-rbac/application-controller/argocd-application-controller-clusterrole.yaml` at the v3.4.4 tag — confirmed `apiGroups:['*'], resources:['*'], verbs:['*']` unchanged. The conclusion (no impact on automated sync) is based on the source, not the changelog summary.
- **`--server-side` fix:** confirmed by running `scripts/20-deploy.sh` from scratch against a fresh cluster and observing all resources `serverside-applied` with no errors.
- **emptyDir necessity:** 30 real `curl http://localhost:8080/api/quote` requests through the Ingress, then full pod log review across all 3 pods. Zero errors. Not inferred from reading uvicorn's source — tested with actual traffic.
- **HPA metrics:** waited for `cpu: X%/70%` (not `<unknown>`) by polling `kubectl get hpa` in a loop. HPA showed `cpu: 39%/70%` during the burst test — confirming both that metrics were live and that the burst registered on the controller.
- **`run-all.sh` from host:** ran `./scripts/run-all.sh` from the host filesystem (not inside the toolbox) and confirmed it reaches `exit 0` through the full deploy path.
- **Idempotency:** second run of `scripts/20-deploy.sh` confirmed "argocd-server deployment found — skipping", Application `unchanged`, same 3 pods, no new resources created.

## Summary

Part 2 (basics pass) produced four genuine AI errors caught through real execution and user challenge — none of which would have been visible from code review alone. Two required actual test runs to surface (the CRD annotation limit error and the emptyDir false justification). One required the user to push back on the version choice (v2.13.3 vs v3.4.4). One required explicitly testing the exact reviewer command path (`./scripts/run-all.sh` from the host) rather than the convenient developer path (from inside the toolbox). The pattern across all four: the AI produced output that was locally plausible but broke on the first contact with the real execution environment or the real reviewer flow.
